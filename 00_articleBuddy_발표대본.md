# articleBuddy 중간발표 대본 (2분)

---

안녕하세요, articleBuddy를 개발하고 있는 [이름]입니다.

**[홈 화면 노출 — 문제 정의 카드]**

최근에 저와 친구가 함께 매일 영어 기사를 읽기로 했습니다. 그런데 막상 시작하니 기사는 따로 찾고, 모르는 단어는 다른 앱에서 검색하고, 인증은 또 카톡으로 하는 파편화된 경험 때문에 결국 작심삼일이 됐습니다.

**[비전 카드로 스크롤]**

articleBuddy는 이 문제를 정면으로 해결하는 올인원 영어 학습 플랫폼입니다. 뉴스뿐 아니라 모든 영어 텍스트를 학습 도구로 바꾸고, 출석 도장 시스템으로 지속 가능한 습관을 만드는 것이 저희 비전입니다.

**[WBS 페이지로 이동]**

일정은 이렇게 상세한 WBS로 체계화했습니다. 칸반보드로 현황을 바로 확인해 보겠습니다.

**[칸반보드 아이콘 클릭]**

현재 기획·설계를 완료하고 핵심 구현 단계에 진입한 상태입니다. 기사 파싱, 단어장 저장, Google 소셜 로그인이 Done으로 넘어왔습니다.

**[캘린더 아이콘 클릭]**

5월부터 계획대로 달려왔고, 6월 15일 최종 완성을 목표로 남은 마일스톤을 하나씩 닫아갈 예정입니다.

**[아키텍처 페이지로 이동]**

기술 스택입니다. 클라이언트는 Expo React Native로 iOS·Android를 단일 코드베이스로 관리했고, 백엔드는 Supabase를 선택해 인증·DB·실시간 구독을 한 플랫폼에서 해결했습니다. 외부 API 선정에도 공을 들였는데요. 단어 뜻과 발음은 API 키 없이 무료로 쓸 수 있는 Free Dictionary API를 채택했고, 난이도 분석은 CEFR 단어 목록을 로컬에서 직접 처리해 외부 호출 비용을 제로로 만들었습니다. AI 리포트 생성에는 월 100만 토큰 무료 한도를 제공하는 Gemini 1.5 Flash를 선택했습니다.

**[시퀀스 다이어그램 가리키며]**

흐름을 보시면, 사용자가 기사 URL을 입력하면 서버가 파싱과 난이도 분석을 수행하고, 리뷰를 제출하는 순간 출석과 스트릭이 자동으로 기록됩니다.

**[배치 다이어그램 가리키며]**

전체 시스템은 모바일 앱, Supabase, 외부 API, 이 세 영역으로 깔끔하게 분리되어 있습니다. 특히 눈여겨봐 주셨으면 하는 부분이 있는데요. 외부 API 키가 필요한 Gemini 호출은 반드시 Supabase Edge Function을 통해서만 이루어지도록 설계했습니다. 클라이언트 코드에 API 키가 노출되는 구조를 처음부터 막은 겁니다. 반면 Free Dictionary API와 CEFR 단어 목록처럼 인증이 필요 없는 리소스는 앱에서 직접 호출해 불필요한 서버 왕복을 줄였습니다.

**[ERD 가리키며]**

데이터베이스는 다섯 개 테이블로 구성되어 있습니다. 모든 테이블이 users를 중심으로 연결되는 구조인데, 특히 words 테이블이 user_id와 article_id를 동시에 외래 키로 가지고 있어서 "어떤 사용자가 어떤 기사에서 저장한 단어인지"를 한 번에 추적할 수 있습니다. attendance 테이블은 의도적으로 단순하게 user_id와 날짜만 저장하는데, 스트릭 계산은 DB가 아닌 서버 로직에서 처리하도록 역할을 분리한 결과입니다.

**[마무리]**

이상이 articleBuddy의 현재 설계입니다. 남은 3주 안에 친구 기능과 AI 리포트를 마무리해 완성된 서비스로 찾아오겠습니다. 감사합니다.
